# Proof-Outliner Rules

ALWAYS: Read all explorer reports before outlining — they contain verified computations and distinct openings (round 1)
ALWAYS: Create approaches with genuinely different framings, not just technique variations on the same idea (round 1)
ALWAYS: Identify the KEY GAP in each approach and state the mechanism that would close it (round 1)
NEVER: Create approaches that share the same bottleneck gap — if they hit the same wall they die together (round 1)
